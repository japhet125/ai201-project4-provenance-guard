# Provenance Guard

## Overview

Provenance Guard is a Flask-based backend system that helps creative platforms estimate whether submitted text is more likely human-written or AI-generated. Rather than making absolute claims, the system combines multiple detection signals into a confidence score, displays a transparency label, records every decision in an audit log, and provides an appeals workflow for creators who disagree with the result.

The system is designed to communicate uncertainty honestly and reduce harmful false-positive classifications.

---

# Architecture Overview

A submitted piece of text follows this workflow:

```text
POST /submit
      │
      ▼
Validate request
      │
      ▼
Signal 1 (Groq LLM)
      │
      ▼
Signal 2 (Stylometric heuristics)
      │
      ▼
Confidence scoring
      │
      ▼
Transparency label
      │
      ▼
Audit log
      │
      ▼
JSON response
```

If a creator disputes the classification:

```text
POST /appeal
      │
      ▼
Update status
      │
      ▼
Store creator reasoning
      │
      ▼
Audit log
      │
      ▼
Return confirmation
```

# API Endpoints

## POST /submit

Accepts submitted text and returns:

* attribution
* confidence score
* transparency label
* signal scores
* content ID

---

## POST /appeal

Accepts:

* content ID
* creator reasoning

Updates the submission status to **under_review** and records the appeal in the audit log.

---

## GET /log

Returns recent audit log entries in JSON format.

# Detection Signals

## Signal 1 — Groq LLM

Measures:

* semantic coherence
* writing style
* generic phrasing
* overall likelihood of AI-generated writing

Strengths

* understands meaning
* evaluates overall writing quality

Limitations

* may classify formal human writing as AI
* may miss heavily edited AI-generated writing

---

## Signal 2 — Stylometric Heuristics

Measures:

* sentence length
* sentence length variance
* type-token ratio
* punctuation density

Strengths

* measurable
* independent from the LLM

Limitations

* poems
* very short text
* academic writing
* non-native English writing

---

# Confidence Scoring

The final confidence score combines the two signals.

Formula:

```text
0.60 × LLM score
+
0.40 × Stylometric score
```

Confidence ranges:

|     Score | Result       |
| --------: | ------------ |
| 0.00–0.35 | Likely Human |
| 0.36–0.69 | Uncertain    |
| 0.70–1.00 | Likely AI    |

This conservative threshold reduces false positives by requiring stronger evidence before labeling content as AI-generated.

# Transparency Labels

## High-confidence AI

> "Provenance Guard found strong signals that this content may have been AI-generated. This label is based on automated analysis and may be appealed by the creator."

---

## High-confidence Human

> "Provenance Guard found strong signals that this content was likely written by a human. This label is based on automated analysis and should not be treated as absolute proof."

---

## Uncertain

> "Provenance Guard could not confidently determine whether this content was written by a human or generated by AI. Readers should interpret this result as uncertain."

---

# Appeals Workflow

Creators may contest a classification by submitting:

* content ID
* explanation

The system:

1. updates the submission status to `under_review`
2. stores the creator's reasoning
3. records the appeal in the audit log

Automatic reclassification is intentionally not performed.

---

# Rate Limiting

Configuration:

```text
10 requests per minute
100 requests per day
```

Reasoning:

Most creators submit only a small number of works. These limits allow normal usage while preventing automated abuse and denial-of-service style request floods.

Testing confirmed that requests exceeding the limit return **HTTP 429 Too Many Requests**.

---

# Audit Log

Every submission records:

* timestamp
* content ID
* creator ID
* attribution
* confidence score
* LLM score
* stylometric score
* transparency label
* submission status

Appeals additionally record:

* creator reasoning
* previous status
* updated status

---

# Known Limitations

The system is designed to estimate likelihood rather than prove authorship.

Known difficult cases include:

* highly edited AI-generated text
* formal academic writing
* poetry
* creative experimental writing
* non-native English writing

These limitations motivated the inclusion of an appeals workflow and an uncertainty category.

# Spec Reflection

The planning document provided a clear blueprint before implementation began. Defining the API endpoints, confidence thresholds, transparency labels, and system architecture first made implementation more organized and reduced redesign later in development.

One implementation change from the original plan was changing the Flask server from port **5000** to **5050** because port 5000 was already occupied by macOS AirPlay (AirTunes), which prevented the API from being reached during testing.
